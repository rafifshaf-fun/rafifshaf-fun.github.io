MLOps Tanpa Burnout

📅 April 2024 ⏱️ 7 menit baca 🏷️ MLOps · Docker · MLflow · Grafana

Setiap tutorial MLOps sepertinya berasumsi bahwa Anda memiliki cluster Kubernetes, tim berisi 5 engineer DevOps, dan budget cloud yang bisa membuat seorang founder startup menangis. Inilah realitas bagi kebanyakan dari kita: sebuah VPS dengan 4 vCPU dan RAM 8GB, Docker Compose, dan tekad untuk membuatnya berhasil.

Saya menjalankan stack MLOps saya dari Garut, Indonesia — bukan persisnya region AWS us-east-1. Berikut ini adalah apa yang saya pelajari tentang membangun infrastruktur ML yang andal tanpa harus merasa kelelahan ekstrim (burnout) atau membakar uang.

Inilah yang benar-benar Anda butuhkan — dan apa yang bisa Anda lewati:

✅ Wajib Punya

  • Experiment tracking (MLflow). Anda PASTI akan lupa hyperparameter mana yang menghasilkan output mana. MLflow itu gratis, bisa berjalan di mana saja, dan hanya butuh 20 menit untuk disiapkan.
  • Containerisasi (Docker). Kalimat "tapi jalan kok di laptop saya" adalah musuh utama dari production ML. Docker Compose menangani seluruh stack Anda dalam satu file docker-compose.yml.
  • Monitoring (Grafana). Jika Anda tidak bisa melihat penyimpangan (drift) dari prediksi, Anda tidak tahu apakah model Anda sedang menurun performanya. Menyiapkan dashboard Grafana butuh waktu satu jam dan akan langsung memberikan keuntungan.
  • Versioning model. MLflow Model Registry bisa menangani ini. Setiap model yang ada di production harus punya versi, status (staging/production/archived), dan asal-usul yang jelas kembali ke proses pelatihannya.

❌ Bisa Dilewati (Untuk Saat Ini)

  • Kubernetes — Docker Compose sudah cukup untuk deployment dalam satu server (single-node)
  • Feature store (Feast, Tecton) — sebuah tabel PostgreSQL dengan kumpulan fitur yang diberi versi sudah berfungsi dengan baik pada awalnya
  • Pipeline CI/CD untuk ML — retraining manual dengan MLflow API tidak masalah saat Anda satu-satunya engineer di sana
  • Versioning data (DVC) — Git LFS atau bahkan folder data yang dinamai dengan tanggal sudah bisa digunakan untuk dataset kecil hingga menengah
  • Framework serving model (TorchServe, Triton) — menggunakan FastAPI sebagai pembungkus model Anda jauh lebih sederhana dan lebih mudah di-debug

Prinsip MLOps untuk Solo Engineer

1. Pilih Teknologi yang Membosankan (Boring Technology)

Pilih PostgreSQL ketimbang database time-series khusus. FastAPI ketimbang TensorFlow Serving. Docker Compose ketimbang Kubernetes. Setiap pilihan teknologi harus bisa dipertahankan dengan alasan: "Saya bisa men-debug ini pada jam 2 pagi ketika semuanya rusak." Jika Anda tidak bisa, berarti itu terlalu rumit.

2. Monitoring Itu Harga Mati

Monitoring minimal yang harus ada:

  • Distribusi prediksi seiring waktu — histogram yang diperbarui setiap hari. Jika bentuknya berubah, data Anda berubah.
  • Penyimpangan fitur (Feature drift / PSI) — Population Stability Index untuk 10 fitur terpenting. Berikan peringatan jika PSI > 0.2.
  • Latensi model p50/p95/p99 — karena model yang lambat sama dengan model yang rusak.
  • Tingkat error — kegagalan prediksi, timeout, output NaN.

Semua ini muat dalam satu dashboard Grafana. Siapkan sekali, dan cukup lihat sekilas setiap hari.

3. Otomatiskan Pemicu Retraining, Bukan Jadwalnya

Melatih ulang model pada jadwal tetap (setiap hari Senin!) adalah pemborosan dan sering kali melewatkan momen kapan retraining sebenarnya dibutuhkan. Alih-alih begitu, piculah retraining hanya saat metrik drift melewati batas aman (threshold). Ini berarti retraining terjadi saat benar-benar dibutuhkan — bukan karena kalender yang menyuruhnya.

4. Dokumentasikan Keputusan Arsitektur

Saat Anda adalah solo engineer, tidak ada orang yang bisa ditanya "kenapa kita memilih XGBoost dibanding LightGBM?" Enam bulan dari sekarang, Anda-lah orang yang akan menanyakan itu. Tulislah. File DECISIONS.md sederhana di repo Anda akan menyelamatkan Anda dari pekerjaan arkeologi yang menghabiskan waktu berjam-jam.

Apa yang Akan Saya Tambahkan Jika Ada Resource Lebih

  • Feature store yang proper ketika feature engineering mulai menjadi bottleneck
  • Shadow deployment untuk rilis model baru dengan risiko nol
  • Infrastruktur A/B testing untuk membandingkan performa model langsung di production
  • Hyperparameter tuning otomatis (Optuna) yang terintegrasi dengan MLflow

Tapi semua itu bukanlah pemblokir utama. Anda tetap bisa merilis sistem ML yang andal hari ini juga menggunakan stack yang dijelaskan di atas. Saya melakukannya setiap hari — dari Garut, menggunakan VPS yang biayanya lebih murah daripada makan malam di luar.

Kesimpulan

MLOps tidak membutuhkan gelar PhD dalam infrastruktur komputer. Mulailah dengan experiment tracking dan monitoring. Tambahkan kompleksitas hanya ketika setup yang sekarang benar-benar mulai menyusahkan. Mayoritas proyek ML tidak gagal karena infrastruktur yang tidak memadai — mereka gagal karena tidak ada yang tahu apakah modelnya masih berfungsi dengan baik.
